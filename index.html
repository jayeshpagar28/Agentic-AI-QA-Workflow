Executive Summary

EcomCircles-V2 is an AI-driven Quality Engineering framework for the EcomCircles e-commerce operations platform — replacing brittle, hand-written test scripts with an agentic pipeline that thinks, tests, repairs, and reports.

🎯Project Overview

A complete QA automation suite that validates the EcomCircles SaaS platform (Dashboard, Settings, Warehouse/Purchase Orders, Authentication and multi-tenant RBAC) running against the live staging environment. AI agents handle the full lifecycle — from interpreting a ClickUp user story to logging a reproducible defect with screenshot evidence.

💼Business Objective

Cut manual regression effort, raise release confidence, and shorten the feedback loop between development and QA. Every acceptance criterion becomes traceable: requirement → test plan → automated test → pass/fail → ClickUp defect, giving stakeholders a continuously auditable quality signal.

🏆Key Achievements

  • 240 automated test cases across 44 scenario groups
  • 5 production modules fully automated end-to-end
  • 3 specialised AI agents: Planner · Generator · Healer
  • 19 reusable Page Object Models, 102 evidence screenshots
  • Zero-flake auth via worker-scoped Supabase session fixture
  • Automated defect filing + sprint sync to ClickUp

AI Agentic Framework Overview

The framework treats QA as an orchestration problem. A coordinator (Claude in-session) decomposes the goal and delegates to three specialised sub-agents, each with a tightly scoped toolset, that collaborate through shared file artifacts.

🧠 Design Philosophy

Separation of concerns by cognition: planning, generation and repair are different reasoning tasks, so each is a dedicated agent with least-privilege tools. Read-only agents (Planner, Generator) can never corrupt the codebase; only the Healer holds write access — and even then, fixes follow strict layering rules (selectors live in Page Objects, never in specs). Humans stay in the loop at decision gates, not in the keystrokes.

🗺️Planner Agent

playwright-test-planner · Claude Sonnet

  • Explores the live app via Playwright MCP
  • Maps user journeys & edge cases
  • Emits a structured markdown test plan
  • Tools: browser + planner (read-only)

⚙️Generator Agent

playwright-test-generator · Claude Sonnet

  • Turns the plan into Playwright code
  • Builds Page Object Models per module
  • Verifies steps live as it writes
  • Tools: browser + generator (no edit)

🩹Healer Agent

playwright-test-healer · Claude Sonnet

  • Runs the suite, debugs one failure at a time
  • Root-causes selector / timing / data issues
  • Fixes the correct layer, never patches specs
  • Only agent with Edit / Write access

🧭Decision-Making Process

The orchestrator reads a user story, extracts entities, state transitions, business rules and integrations, then routes work to the right agent. Decision gates — "is the plan complete?", "did the test pass?", "is this a real defect?" — determine whether to advance, retry, or escalate to a human reviewer.

💾Memory & Context Handling

State is durable and file-based: user-stories/, specs/, tests/, screenshots/ and test-results/ form a shared "blackboard". Persistent CLAUDE.md memory captures hard-won lessons (e.g. the Supabase auth-flake root cause). Worker-scoped fixtures hold live session context in-memory during a run.

🔄Tool Execution Lifecycle

①

Perceive

Snapshot the page / read artifacts via MCP

②

Reason

Select the next tool & arguments

③

Act

Click, type, navigate, write file

④

Observe

Capture result, console & network

⑤

Reflect

Verify outcome, retry or advance

Decision Points

Agent selection, validation pass/fail, and "is it fixable?" gate the flow and prevent wasted cycles.

Retry / Self-Heal

Failures route to the Healer, which fixes the right layer and re-runs — bounded by a 1-retry safety net for transient staging blips.

Parallel Paths

Specs run fully parallel across 3 browser engines with 2 workers, respecting Supabase login rate limits.

Repository Analysis

A clean, artifact-driven layout where each folder is one stage of the pipeline — inputs flow left to right into automation, evidence and reports.

🗂️Folder Structure

📁 prompts/ — QA prompt templates (story → plan)
📁 user-stories/ — stories synced from ClickUp Docs
📁 specs/ — generated test plans
📁 tests/ — Playwright suites + Page Objects
↳ fixtures/ · pages/ · utils/ · {STORY}/
📁 scripts/clickup/ — sync & reporting CLIs
📁 screenshots/ — pass/fail evidence by module
📁 test-results/ · reports/ — MD & XLSX outputs
📄 playwright.config.ts · .mcp.json · .env

🧱Design Patterns

  • Page Object Model — locators & actions encapsulated per module (19 POMs)
  • Worker-scoped auth fixture — one login per worker, in-memory Supabase session
  • Role-based test factories — typed fixtures per RBAC role
  • Resilient navigation — retries SPA hydration bounce, no fixed sleeps
  • Severity → Priority mapping — standardized defect risk model
  • Defect auto-classification — keyword heuristics (API/UI/Functional…)

♻️Reusable Components

  • SidebarNavigation & AccountSwitcher (shared/)
  • MailinatorClient — invite-email polling
  • gotoAuthed() — hydration-safe navigation util
  • ClickUp API wrapper — tasks, docs, attachments
  • parse-report.ts — markdown defect-log parser

🔗Dependency Relationships

Specs depend on Page Objects + role fixtures; fixtures depend on Supabase auth and utils; the ClickUp CLI depends on the API wrapper + report parser. The app under test is decoupled behind baseURL, so the same suite targets any environment.

Testing Framework Analysis

An 8-step AI-driven pipeline takes a story from prompt to logged defect, blending exploratory heuristics, generated automation, and self-healing execution.

🧪Strategy & Execution Flow

Step 1–2

Analyse & Plan

Story decomposed into entities, states & rules → test plan across 10 test types.

Step 3

Exploratory Testing

Live MCP browser session applies QA heuristics (double-submit, loader clicks, persistence).

Step 4–5

Generate & Heal

POM automation generated, executed, and auto-repaired until green.

Step 6–8

Report & Sync

QA report → defect log parsed → ClickUp parent + subtasks with screenshots.

✅Validation & Reporting

Scenario generation: 10 test types — Happy Path, Negative/Validation, State-Aware UI, Full CRUD, Business-Rule, Field Deep-Validation, UI Persistence, Audit/Data, Third-Party, Async/Race.

Validation: Playwright assertions + network status scanning (flags 4xx/5xx). Reporting: per-AC coverage matrix, pass/fail/blocked stats, mandatory defect-log table parsed automatically.

Severity → Priority: Critical→Urgent · High→High · Medium→Normal · Low→Low, mapped straight into ClickUp.

Future Roadmap

A staged evolution from today's three-agent pipeline toward a fully autonomous, predictive quality platform.

🌅Near Term

  • Multi-Agent Collaboration — concurrent specialised crews
  • MCP Servers — first-class ClickUp, Stripe, DB connectors
  • Self-Healing 2.0 — broader auto-repair coverage
  • Autonomous Test Generation — net-new stories → suites

🌄Mid Term

  • RAG Architecture — retrieve specs, defects & docs at query time
  • Long-Term Memory — vector store of failure patterns
  • AI Bug Classification — severity & ownership prediction
  • CI/CD Intelligence — risk-based test selection per PR

🌇Long Term

  • Predictive Failure Analysis — flag fragile areas pre-release
  • Cloud Scaling — elastic parallel grids
  • Voice & Chat Interfaces — "test the checkout flow" on demand

🧭 Guiding Principle

Each step keeps humans in control at decision gates while progressively removing them from repetitive keystrokes — the system grows more autonomous without growing less accountable.
