The Problem

Two failure modes that break automation

🎲Flaky Tests

Tests that pass and fail without a code change destroy trust. In a real-world case the root cause was authentication: a token-based auth provider stored sessions in localStorage and rotated refresh tokens on every refresh, so reusing a saved storageState file went stale and failed intermittently — plus concurrent logins hit rate limits.

⏳Long-Running Agents

An AI agent generating and healing hundreds of tests can run for a long time. Without help it loses earlier context, repeats work, drifts off task, or dies mid-run — wasting tokens and producing inconsistent output.

🎯 The goal

Make every run deterministic (no flakiness), durable (survives long autonomous sessions), and accurate (results you can publish to stakeholders).

Solution 1 · Reliability

How flaky tests are eliminated

🔑Worker-scoped auth

Each worker logs in once into a live context kept alive for its lifetime — tokens refresh in memory, never from a stale file.

⏱️Backoff & stagger

Staggered worker startup + exponential backoff absorbs Supabase concurrent-login rate limits.

🧭No blind waits

gotoAuthed() retries the SPA hydration bounce; zero waitForTimeout() — waits are tied to real signals.

Result: the suite passes at retries = 0; the single configured retry is a safety net, not a flake mask.

🪟1M context

Opus 4.8's large window holds the whole task — plan, code and failures — in view at once.

🗜️Context management

When context grows, the harness summarizes and carries state forward so work continues seamlessly.

💾Persistent memory

Hard-won lessons (e.g. the auth-flake fix) are written to memory and recalled in future runs.
